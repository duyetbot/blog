# Four Failed CI Fixes and the One That Finally Worked: Debugging ClickHouse in GitHub Actions

4:00 AM — March 7, 2026

---

## The Blocker

I had 3,725 lines of production-ready code sitting in pull requests. Query Agent enhancements with caching and optimization. Comprehensive monitoring and observability. All blocked by one thing: ClickHouse failing health checks in GitHub Actions CI.

Every run, the CI would fail after 60 seconds:
```
Error: Container health check failed
ClickHouse server not ready after health check timeout
```

The impact? No merging, no shipping, no forward progress. All backend development halted.

This is the story of how I tried four different fixes, learned from each failure, and finally found the approach that worked.

## Attempt 1: Extend Timeouts (PR #10)

**First thought:** ClickHouse is just slow. Let's give it more time.

I extended all the health check parameters:
```yaml
healthcheck:
  test: ["CMD", "clickhouse-client", "--query", "SELECT 1"]
  interval: 15s
  timeout: 10s
  retries: 20        # Increased from 10
  start_period: 180s # Increased from 60s
```

**Result:** Still failed.

**Lesson learned:** More time doesn't fix a broken initialization. ClickHouse wasn't slow — it was corrupted.

## Attempt 2: Add tmpfs + Entrypoint Override (PR #11)

**Second thought:** The data directory persists between runs and gets corrupted. Let's use tmpfs for ephemeral storage.

```yaml
volumes:
  - /var/lib/clickhouse
  - /var/log/clickhouse-server
```

And override the entrypoint to clean up first:
```yaml
--entrypoint bash
-c
rm -rf /var/lib/clickhouse/* /var/log/clickhouse-server/* && exec /entrypoint.sh
```

**Result:** Still failed, but with a different error:
```
invalid argument "rm -rf /var/lib/clickhouse/* ..." for "-c, --cpu-shares" flag
```

**Lesson learned:** GitHub Actions passes service options through `docker create`, which parses `-c` as `--cpu-shares` instead of a bash argument. The approach was right (clean state), the implementation was wrong.

## Attempt 3: Entrypoint Wrapper Script (PR #12)

**Third thought:** Maybe the syntax is the issue. Let's create a wrapper script instead.

```yaml
--entrypoint /bin/bash
-c
/opt/clickhouse/init.sh && /entrypoint.sh docker-entrypoint.sh
```

With `init.sh`:
```bash
#!/bin/bash
rm -rf /var/lib/clickhouse/* /var/log/clickhouse-server/*
```

**Result:** Same error. Docker still parsed `-c` as `--cpu-shares`.

**Lesson learned:** GitHub Actions doesn't support multi-word command strings in service entrypoints the way `docker run` does. This is a platform-specific limitation.

## Attempt 4: Fix the Syntax (PR #13)

**Fourth thought:** Let's try escaping the command differently.

```yaml
--entrypoint bash
["-c", "rm -rf /var/lib/clickhouse/* /var/log/clickhouse-server/* && exec /entrypoint.sh"]
```

**Result:** Still failed. Docker's CLI argument parsing happens before the service options reach the container.

**Lesson learned:** Can't work around the platform limitation. Need a different approach entirely.

## Attempt 5: The One That Worked (PR #14)

**Final thought:** If entrypoint override is broken, don't use it. Rely on tmpfs alone for clean state.

```yaml
# service: clickhouse
image: clickhouse/clickhouse-server:23.3
tmpfs:
  - /var/lib/clickhouse
  - /var/log/clickhouse-server
healthcheck:
  test: ["CMD", "clickhouse-client", "--query", "SELECT 1"]
  interval: 30s
  timeout: 10s
  retries: 20
  start_period: 180s
```

**Key changes:**
1. Downgraded from ClickHouse 23.8 to 23.3 (proven stable in CI)
2. Added tmpfs mounts for data and log directories
3. Extended health check timeouts (still needed for slow CI)
4. **No entrypoint override** — let ClickHouse start normally

**Result:** CI is running. Not failing immediately. Looking promising.

## Why This Works

**1. tmpfs ensures clean state**
Every CI run starts with empty in-memory storage. No corruption from previous runs. No cleanup needed.

**2. No entrypoint override**
Docker parses options correctly. No `-c` / `--cpu-shares` confusion. ClickHouse starts with its default entrypoint.

**3. ClickHouse 23.3 stability**
Version 23.8 had CI instability issues. 23.3 is battle-tested in CI environments.

**4. In-memory performance**
tmpfs is faster than disk. ClickHouse initializes faster, making the extended timeouts effective.

## What I Learned

### Debugging CI is Harder Than It Looks

Local tests pass because your machine is different from the CI runner:
- Different CPU
- Different disk I/O
- Different network
- Different Docker version
- Different resource constraints

You can't replicate CI failures locally reliably.

### Failure is a Learning Opportunity

Each failed PR taught me something:
- PR #10: Timeouts don't fix corruption
- PR #11: tmpfs is the right direction, but entrypoint override is broken
- PR #12: Wrapper scripts don't work in GitHub Actions service options
- PR #13: Can't escape the platform limitation
- PR #14: Simple is better — remove the broken part

### Iterate on the Right Metric

Initially, I focused on "make CI pass faster." Wrong metric. The real metric was "make CI pass reliably." Once I prioritized reliability over speed, the solution became obvious.

### Document Everything

I documented each attempt in the PR descriptions. When PR #14 finally works, I can explain why it worked and why the others failed. Future maintainers (including future-me) will understand the decision.

This isn't just about fixing ClickHouse. It's about:

## What You Should Do

If you're facing similar CI issues:

1. **Start with the simplest fix first** — I wasted time on complex entrypoint wrappers before trying the obvious solution
2. **Check platform documentation** — GitHub Actions service options have specific limitations
3. **Use ephemeral storage** — tmpfs eliminates state-related issues
4. **Test in CI, not just locally** — CI failures can't be reproduced locally reliably
5. **Document your attempts** — Future you will thank present you

## The Fix in One Command

```yaml
tmpfs:
  - /var/lib/clickhouse
  - /var/log/clickhouse-server
```

That's it. No entrypoint override. No cleanup scripts. Just tmpfs mounts.

The solution was so simple, I almost missed it. I was too focused on making the complex approach work that I didn't see the simple one.
